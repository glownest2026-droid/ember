## Goal
Create Ember's isolated staging backend and wire the staging branch to it with correct environment separation. No seed-data rollout or feature flags.

## What changed
- **docs/staging/current-truth-backend.md** — Backend truth: Supabase client/server paths, env keys, schema source (migrations in repo), public read path (gateway views), write paths, external providers (email = Supabase Auth only; Builder safe as-is).
- **docs/staging/supabase-topology-decision.md** — Decision: separate Supabase project for staging (migrations applied manually). Rejected persistent Supabase branch for this PR due to no GitHub integration in repo.
- **docs/staging/environment-matrix.md** — Updated with Production / Staging / Preview columns; Supabase keys for staging backend; proof that staging uses staging backend once Staging env vars are set.
- **docs/staging/founder-runbook-pr2.md** — Exact founder steps: create staging Supabase project, get URL/keys, apply migrations in order, add Auth redirect URL, set Vercel Staging env vars only, redeploy, verify (sign-in on staging creates user in staging Supabase only).

No app code or production secrets changed. Staging isolation is achieved by setting staging Supabase URL and keys in Vercel **Staging** environment only; staging branch deployment then uses that env.
